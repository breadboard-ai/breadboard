# Project Rewind ‚Äî Session Rollback

Bees agents accumulate long, expensive conversation histories. When an agent
goes off-track at turn 12 of a 20-turn session, the only option today is to
delete the task and start over ‚Äî losing all prior context and token investment.

This project adds a **rollback** capability: rewind a session to any prior turn
boundary and resume from there. The UI surfaces a rollback button on each turn
in hivetool's session detail view.

## The Session Lineage Model

A **task** owns one or more **sessions**. Exactly one session is **active** ‚Äî
the one the agent is currently running or suspended in. Past sessions are
**superseded**: intact, inspectable, never mutated.

Session IDs are UUIDs ‚Äî the same ones already generated by `GeminiRunner`
(`str(uuid.uuid4())`) and stored in `InteractionState.session_id`. The session
directory name is the full UUID. (Examples below use short prefixes for
readability.)

```
Task "Chat 4" (ticket 1b5f5804-...)
  ‚îú‚îÄ‚îÄ Session eca495b3-... (superseded)
  ‚îÇ     forked at turn 3 ‚Üí a7c21f09-...
  ‚îÇ     12 turns, 339 context entries, fully preserved
  ‚îÇ
  ‚îú‚îÄ‚îÄ Session a7c21f09-... (superseded)
  ‚îÇ     forked from eca495b3-... at turn 3
  ‚îÇ     forked at turn 7 ‚Üí d4e88a1c-...
  ‚îÇ     8 turns, fully preserved
  ‚îÇ
  ‚îî‚îÄ‚îÄ Session d4e88a1c-... (active, suspended)
        forked from a7c21f09-... at turn 7
        5 turns, waiting for input
```

**Rollback = fork.** When the user says "rewind to turn 3", the system:

1. Creates a **new** session with the conversation context truncated to turn 3
   and the **filesystem snapshot** from that turn boundary.
2. Marks the old session as **superseded** (records the fork point).
3. Sets the new session as the task's **active** session.
4. The old session is never modified ‚Äî its events, interaction state, filesystem
   snapshots, and turn boundaries remain intact for inspection.

**Re-activation is free.** Since superseded sessions are preserved, swapping
`active_session` back to an older session restores it without reconstruction.

### Session state

A session owns **all** the state the agent accumulates:

- **Conversation context** ‚Äî the `InteractionState.contents` array.
- **Filesystem** ‚Äî the `InteractionState.file_system` snapshot
  (`FileSystemSnapshot`). This is the agent's workspace: surface.json, bundles,
  data files, everything the agent wrote.
- **Turn boundaries** ‚Äî context offsets recorded at each API call boundary.
- **Interaction state** ‚Äî the full `InteractionState` at suspend/resume points
  (including `function_call_part`, `task_tree`, `consents_granted`).

The filesystem is **not** a separate artifact living at
`tickets/{id}/filesystem/`. It's part of the session. Each session carries its
own filesystem snapshot. When the task switches to a different session (fork,
re-activation), the runtime filesystem is hydrated from that session's snapshot.

### Data model

```
tickets/{task-id}/
  metadata.json
    active_session: "d4e88a1c-..."

  sessions/
    eca495b3-.../                    ‚Üê superseded, untouched
      status                        ‚Üê "superseded"
      events.jsonl
      interaction.json              ‚Üê includes file_system snapshot
      turns.json
      lineage.json                  ‚Üê { forked_to: { session: "a7c21f09-...", at_turn: 3 } }

    a7c21f09-.../                    ‚Üê superseded, untouched
      status
      events.jsonl
      interaction.json
      turns.json
      lineage.json                  ‚Üê { forked_from: { session: "eca495b3-...", at_turn: 3 },
                                         forked_to:   { session: "d4e88a1c-...", at_turn: 7 } }

    d4e88a1c-.../                    ‚Üê active
      status                        ‚Üê "suspended"
      events.jsonl
      interaction.json
      turns.json
      lineage.json                  ‚Üê { forked_from: { session: "a7c21f09-...", at_turn: 7 } }
```

Each session directory is a self-contained `FileBasedSessionStore`. The store
manages one session's lifecycle ‚Äî it doesn't know about tasks or lineage.
Lineage is task-level metadata managed by bees.

### Why fork, not rewind?

- **Non-destructive.** You never lose data. If the fork goes worse, the old
  session is right there.
- **No surgery on live state.** The old session's `InteractionState` is never
  truncated. You copy the relevant prefix into a new session.
- **Explicit lineage.** Each session knows where it came from. Hivetool can
  render the fork tree.
- **Re-activation for free.** Want the old conversation back? Swap
  `active_session`. No reconstruction.

## The Yak

The rollback feature needs structured access to session state ‚Äî the conversation
context, filesystem, turn boundaries, and the ability to truncate and restart.
But bees currently treats session state as **opaque bytes**. It manually
serializes an `InteractionState` blob to `session_state.json`, writes its own
eval log files via `EvalCollector`, and reconstructs session history from
observation artifacts rather than a proper data model.

Meanwhile, opal-backend already defines the right abstractions: `SessionStore`
(lifecycle, event log, interaction snapshots) and `InteractionStore`
(suspend/resume state). In the production backend, these are database-backed.
But `GeminiRunner` creates throwaway `InMemorySessionStore` and
`InMemoryInteractionStore` instances per run ‚Äî the stores evaporate when the run
ends, and bees reimplements persistence outside the abstraction.

The yak: **give bees a persistent `SessionStore`** so session state is properly
managed, not manually extracted and reassembled. Then rollback becomes a fork
operation on the store rather than archaeology on log files.

### The Filesystem Gap

`InteractionState` already has a `file_system: FileSystemSnapshot | None` field.
In production (in-memory filesystem), this captures all file contents as part of
the interaction state ‚Äî it works. In bees (`DiskFileSystem`), the field is
`None`: "the disk is the state." The filesystem lives outside the session
abstraction at `tickets/{id}/filesystem/`.

This is the same leak as the session store. Bees bypasses the abstraction and
then can't do session-level operations on the filesystem. If the user forks a
session at turn 3, the filesystem still contains files written during turns 4‚Äì12
‚Äî inconsistent with the rolled-back context.

The fix: **the filesystem state belongs in the session.** `DiskFileSystem`
continues to serve as the runtime I/O surface, but the session store captures
`FileSystemSnapshot` at turn boundaries. When forking, the new session's
filesystem is hydrated from the snapshot at the fork point. This doesn't tie the
filesystem to being a physical filesystem ‚Äî `FileSystemSnapshot` is abstract
(`dict[str, FileDescriptor]`), compatible with any backing store.

## Phase 1 ‚Äî FileBasedSessionStore ‚úÖ

### üéØ Objective

`GeminiRunner` uses a file-backed `SessionStore` and `InteractionStore` instead
of creating ephemeral in-memory instances. Session events, interaction state,
and **filesystem snapshots** persist to the ticket directory under
`sessions/{session-id}/`. The manual `save_resume_state()` /
`load_resume_state()` dance and the standalone `filesystem/` directory are
eliminated ‚Äî the session store owns all session state.

**Observable proof:** Run a task, let it suspend. Kill the box. Restart the box.
The task resumes successfully ‚Äî the resume state and filesystem were loaded from
the session store. Inspect `tickets/{id}/sessions/` and see the session
directory with status, events.jsonl, interaction.json (containing a
`file_system` snapshot), and the runtime workspace hydrated from it.

### Changes

#### opal-backend

- [x] `FileBasedSessionStore` ‚Äî new `SessionStore` implementation in
      `opal_backend/sessions/file_store.py`. Implements **both** the
      `SessionStore` and `InteractionStore` protocols ‚Äî in the file-based path,
      there's no reason for two separate stores mapping to the same directory.
      Storage layout under a configurable root:
  - `status` ‚Äî plain text file (the `SessionStatus` value)
  - `events.jsonl` ‚Äî append-only, one JSON object per line
  - `interaction.json` ‚Äî durable `InteractionState.to_dict()` snapshot
    (includes `file_system: FileSystemSnapshot`). Persists for inspection.
  - `resume_id` ‚Äî plain text file
- [x] **Pop semantics resolution**: `SessionStore.save_interaction()` writes the
      durable `interaction.json` (persists forever for inspection). The
      `InteractionStore.load()` pop semantics are handled via a separate
      ephemeral marker ‚Äî `load()` clears the `resume_id` file (marking the
      resume as consumed) but does **not** delete `interaction.json`. This
      preserves both the protocol contract and the durability requirement.
- [x] Add `SUPERSEDED = "superseded"` to the `SessionStatus` enum in
      `store.py`. This is a protocol-level addition.
- [x] Tests following `tests/test_session_store.py` pattern.

#### bees ‚Äî session store wiring

- [x] `GeminiRunner` ‚Äî accept store factories instead of hardcoding
      `InMemorySessionStore()`. Default to in-memory for backward compat.
- [x] `SessionConfiguration` ‚Äî add optional `session_id: str | None` field.
      When provided, `GeminiRunner.run()` uses this ID instead of generating
      a new UUID. Required for the fork code path (Phase 4), where the mutation
      handler pre-creates the session directory with a known ID.
- [x] `TaskRunner` ‚Äî provide file-backed store factories rooted at
      `tickets/{id}/sessions/{session-id}/`. Track the active session ID in task
      metadata.
- [x] Task metadata ‚Äî add `active_session` field to `TicketMetadata`. First run
      generates a session ID; subsequent resumes reuse it.
- [x] Remove `save_resume_state()` / `load_resume_state()` /
      `clear_resume_state()` from `bees/session.py`. The file-based store
      handles this.
- [x] `resume_task()` ‚Äî load resume state from the file-based store instead of
      reading `session_state.json`. The `GeminiRunner.resume()` method reads
      from the interaction store directly rather than receiving opaque bytes.
- [x] Migration: handle existing `session_state.json` files gracefully (convert
      on first access or fall back).

#### bees ‚Äî filesystem as session state

- [x] `DiskFileSystem` ‚Äî capture `FileSystemSnapshot` into the
      `InteractionState` at suspend points. The `file_system` field is no longer
      `None` for disk-backed systems.
- [x] On resume, hydrate the `DiskFileSystem` from the `InteractionState`s
      `file_system` snapshot instead of relying on leftover disk state.
- [x] Rework `Ticket.fs_dir` ‚Äî the runtime workspace moves from
      `tickets/{id}/filesystem/` to `tickets/{id}/sessions/{session-id}/workspace/`.
      On session switch (fork, re-activation), the workspace is hydrated from
      the new session's `FileSystemSnapshot`.
- [x] **Subtask filesystem sharing**: The current `owning_task_id` pattern
      (ticket.py:145) lets subtasks share a parent's `filesystem/`. With
      per-session filesystem ownership, subtasks need to share the parent
      _session's_ workspace, not just the parent task's directory. Fix:
      subtasks receive the parent's `active_session` workspace path, not a
      task-level `fs_dir`.
- [ ] Content-addressed deduplication (stretch): for sessions with large files
      (bundles, images), store file blobs in a shared content-addressed store
      and reference by hash in `FileDescriptor`. This keeps snapshots small when
      files don't change between turns. Can be deferred if snapshot sizes are
      manageable without it.

> [!NOTE]
> Phase 1 stores the full `FileSystemSnapshot` at each suspend point (not per
> turn). For a session that suspends 10 times with a 110 KB filesystem, that's
> ~1.1 MB of inline data. This is acceptable. Per-turn lazy snapshotting in
> Phase 3 addresses the higher-frequency checkpoint storage.

shows its session lineage ‚Äî the active session and any superseded ancestors.

## Phase 3 ‚Äî Turn-Aware Session History

### üéØ Objective

The `FileBasedSessionStore` records turn boundaries as first-class data. Given a
session, you can ask "what was the conversation context and filesystem at the
start of turn N?" without parsing eval logs. Hivetool displays the turn
boundaries.

**Observable proof:** After a multi-turn session suspends, open it in hivetool.
The session inspector shows turn boundaries with context offsets and filesystem
snapshot sizes. Each boundary aligns with the turn headers in the existing log
detail view.

### Turn boundary = checkpoint

Each turn boundary is a **checkpoint** ‚Äî an atomic snapshot of both conversation
context and filesystem state at that moment. The two are consistent: the model's
decisions in turns 0..N-1 produced those files, and the context reflects those
decisions.

#### Lazy snapshotting

Most turns don't touch the filesystem ‚Äî they're conversation, API calls,
thinking. In a 90-turn session, maybe 10‚Äì15 turns actually write files. Naively
snapshotting the full filesystem at every turn boundary wastes storage (110 KB √ó
90 turns = ~10 MB for one session).

Instead: **snapshot only when the filesystem changed.** The event tee detects
whether the preceding turn produced file writes (via `CompleteEvent.result` file
entries or `functionResponse` from file-writing tools). If no files changed, the
checkpoint carries `"file_system": null` ‚Äî meaning "same as the previous
checkpoint's filesystem." The fork handler walks backward to find the nearest
non-null snapshot.

`turns.json` with lazy snapshotting:

```json
[
  { "turn": 0, "context_length": 288, "file_system": null },
  { "turn": 1, "context_length": 293, "file_system": null },
  { "turn": 2, "context_length": 310, "file_system": { "files": { ... }, "routes": {}, "file_count": 5 } },
  { "turn": 3, "context_length": 340, "file_system": null },
  { "turn": 4, "context_length": 370, "file_system": null },
  { "turn": 5, "context_length": 395, "file_system": { "files": { ... }, "routes": {}, "file_count": 8 } }
]
```

Forking at turn 4: context truncated to `contents[:370]`, filesystem restored
from turn 2's snapshot (the nearest preceding non-null entry).

The suspend-point `interaction.json` always carries the _latest_ full snapshot
(it's the resume point ‚Äî must be complete). The `turns.json` entries are the
_historical_ checkpoints for rollback. The fork handler reads `turns[N]`, walks
back for the filesystem if needed, and has everything to create the new session.

### Changes

- [ ] Extend `SessionStore` protocol with turn-boundary tracking:
  - `record_turn_boundary(session_id, context_length, file_system_snapshot)` ‚Äî
    captures both the context offset and a `FileSystemSnapshot`.
  - `get_turn_boundaries(session_id) -> list[TurnCheckpoint]` ‚Äî returns the
    recorded checkpoints.
- [ ] `FileBasedSessionStore` ‚Äî persists checkpoints to `turns.json`.
- [ ] **Turn recording lives in bees**, not in opal-backend's `_tee_events()`.
      `_tee_events()` is a generic opal-backend function with no knowledge of
      `DiskFileSystem` or bees-specific file-writing detection. Instead,
      `drain_session()` (or a new callback on `EvalCollector`) records turn
      boundaries into the store when it sees `sendRequest` events. This keeps
      bees concerns in bees and avoids coupling opal-backend to filesystem
      snapshotting.
- [ ] Lazy snapshot detection: the recorder checks whether the preceding turn
      produced file writes (via `functionResponse` from file-writing tools).
      If no files changed, the checkpoint carries `file_system: null`.
- [ ] Hivetool ‚Äî display turn checkpoints in the session inspector. Show context
      offset, filesystem file count, and (if available) token metadata per turn.
- [ ] `EvalCollector` ‚Äî delegate turn tracking to the store instead of
      maintaining its own `turns` list. The store is authoritative.

## Phase 4 ‚Äî Rollback

### üéØ Objective

A `rollback-to-turn` mutation forks a session at a prior turn. A new session is
created with the truncated context, the old session is marked superseded, and
the task resumes from the fork point. Hivetool surfaces a ‚è™ button on each
turn.

**Fork semantics**: "Fork at turn N" means **keep turns 0..N-1, redo from N**.
The context is truncated to `contents[:turn_boundaries[N]]` ‚Äî the state as it
was when turn N _started_, before the model responded. The filesystem is
restored to the snapshot at that boundary. The model will re-generate turn N
and everything after with fresh output.

**Observable proof:** Run a task through 5 turns. It suspends. In hivetool,
click ‚è™ on turn 2. The old session is marked superseded. A new session appears
as active. The box picks up the task and runs it. The new session shows turns
0‚Äì1 as inherited context and fresh output from turn 2. Both sessions are visible
and inspectable in hivetool.

### Changes

#### bees

- [ ] New hot mutation type: `rollback-to-turn`
  - Payload: `{type: "rollback-to-turn", task_id: str, turn_index: int}`
  - Handler in `mutations.py`:
    0. **Guard**: reject if `metadata.status != "suspended"`. Rollback on a
       running task would corrupt state. Only suspended tasks can be forked.
    1. Load the active session's `InteractionState` from the store.
    2. Load turn boundaries and filesystem snapshots from the store.
    3. Generate a new session UUID. Create its directory under
       `tickets/{id}/sessions/`.
    4. Copy `interaction_state.contents[:turn_boundaries[turn_index]]` into the
       new session's `InteractionState`.
    5. Copy the `file_system` snapshot from the fork-point turn boundary into
       the new session's `InteractionState` (walk back for nearest non-null).
    6. Clear `function_call_part` in the new session (no pending suspend).
    7. Write `lineage.json` for both sessions (fork point + parent/child).
    8. Mark old session status as `SUPERSEDED`.
    9. Update task metadata: `active_session` ‚Üí new session ID.
    10. Reset task metadata: status ‚Üí `available`, clear `suspend_event`, adjust
        `turns` count, clear `assignee`.
    11. Hydrate the runtime `DiskFileSystem` from the fork-point snapshot.
    12. Trim `chat_log.json` to match.
    13. Optionally clean up child tasks spawned after the rollback point.
  - The scheduler picks up the task. `TaskRunner` calls `GeminiRunner.run()`
    with `session_id` set to the new UUID (via `SessionConfiguration`).
  - **`seed_files` hazard**: After rollback, the task goes to `available`
    and hits the `run_task()` code path (not `resume_task()`). `run_task()`
    calls `provision_session(seed_files=True)`, which re-seeds skill files
    into the workspace ‚Äî potentially overwriting files the agent modified
    during earlier turns. The fork path must pass `seed_files=False`, just
    like `resume_task()` does. Without this, rollback silently corrupts
    the workspace for any session where the agent modified skill-seeded
    files.

#### GeminiRunner

- [ ] `run()` ‚Äî when the store already contains an `InteractionState` (seeded by
      the fork), use its `contents` as the initial context and hydrate the
      `DiskFileSystem` from its `file_system` snapshot. This is the "fork" code
      path.

#### hivetool UI

- [ ] `log-detail.ts` ‚Äî add a ‚è™ button in `renderTurnHeader()`. On click,
      dispatch a `rollback-to-turn` mutation with the absolute turn index.
- [ ] Confirm dialog: "Fork at turn N? A new session will be created and turns
      N+1 through M will be preserved in the superseded session."
- [ ] Disable rollback buttons when the box is not active (no mutation
      processing).
- [ ] Visual feedback: session lineage updates to show the new active session
      and the superseded ancestor.
- [ ] Wire `rollback-to-turn` mutation type in `mutation-client.ts`.

## Non-Goals

- **Content-addressed blob storage**: Full deduplication of file contents across
  snapshots via content-addressing. The initial implementation stores
  `FileSystemSnapshot` inline in `interaction.json`. If snapshot sizes become
  problematic, a shared blob store with hash references is a future
  optimization.
- **Branching / forking UI**: This project supports linear fork chains (rewind
  to a point, continue from there). A full branching UI where you maintain
  multiple active sessions in parallel is out of scope.
- **Re-activation UI**: The data model supports re-activating a superseded
  session (swap `active_session` back), but the UI for this is out of scope. The
  capability exists for manual use or a future project.
- **Eval log replacement**: The eval log files (`logs/`) continue to be written
  for backward compatibility. The store becomes authoritative for session data,
  but the eval log remains as a secondary record until fully deprecated.
- **Production backend migration**: This project targets the bees/local
  execution path. The production `SessionStore` (database-backed) already
  persists correctly ‚Äî no changes needed there.
